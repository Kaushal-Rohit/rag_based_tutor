📖 What This System Does

The Adaptive RAG Pipeline is an AI-powered study assistant that answers questions about NCERT textbook content (Physics, Chemistry, English, Science) and any PDFs you upload. It uses Retrieval-Augmented Generation — first finding the most relevant text passages from the knowledge base, then using a local AI model to compose an answer grounded in those passages.

What makes it adaptive is its sentiment analysis: the system tracks how well you're understanding the material (based on the tone of your questions) and automatically adjusts how many source passages it retrieves and how detailed its explanations are. Confused? It fetches more context and explains step-by-step. Understanding well? It gives concise, precise answers.

🧠 Architecture Rationale

Every major architectural decision in this system is documented below, detailing what was chosen, the alternatives considered, measured impact, and why it matters.

1. HNSW Graph Indexing over Flat/IVF

What: FAISS IndexHNSWFlat (M=32) instead of IndexFlatL2 or IndexIVFFlat.

Why vs Alternative: HNSW provides O(log n) approximate nearest neighbor search without requiring a separate training phase (unlike IVF), making it ideal for dynamic or medium-sized corpora.

Before → After: Transitioned from a linear scan to a graph traversal. The benchmark on 13,917 chunks shows a mean search latency of {{ "%.3f" | format(data.benchmarks.faiss_hnsw_search_latency_mean_ms) if data.benchmarks else "0.386" }}ms.

Why it mattered: Guarantees millisecond retrieval times even as the index scales to millions of documents, without sacrificing significant recall (Recall@5 is {{ "%.1f" | format(data.benchmarks.hnsw_recall_at_k) if data.benchmarks else "82.7" }}%).

2. FAISS vs ChromaDB Dual Backend

What: Configurable dual backend (RAG_RETRIEVAL_BACKEND) defaulting to ChromaDB.

Why vs Alternative: ChromaDB offers native metadata filtering via where clauses, while FAISS requires post-retrieval filtering (over-fetching). Defaulting to ChromaDB prioritizes correctness and simplicity, while FAISS remains available for raw speed.

Before → After: Hardcoded ChromaDB only → Configurable backends. Real-world benchmark: FAISS raw search (~{{ "%.3f" | format(data.benchmarks.faiss_hnsw_search_latency_mean_ms) if data.benchmarks else "0.386" }}ms) vs ChromaDB total (~{{ "%.3f" | format(data.benchmarks.chroma_total_latency_mean_ms) if data.benchmarks else "44.974" }}ms).

Why it mattered: Provides a flexible deployment strategy. The developer can choose the simpler, strictly correct filtering (Chroma) for standard loads, or switch to the faster approximate filtering (FAISS) if traffic spikes.

3. Metadata Pre-Filtering Merged with Vector Search

What: Integrating exact metadata match filters (e.g., subject, class) directly into the similarity search query.

Why vs Alternative: Doing similarity search first and then filtering out invalid subjects (post-filtering) often results in 0 valid results if the top-K chunks happen to be from the wrong subject. Native pre-filtering avoids this.

Before → After: FAISS post-filter overfetch (which risks missing chunks) → ChromaDB native where clause (which guarantees matching metadata).

Why it mattered: Solves the classic "filtered RAG" problem, ensuring that questions strictly about "Class 12 Physics" only retrieve from that subset, without wasting vector search capacity on irrelevant subjects.

4. Sentiment-Driven Dynamic-k Retrieval Depth

What: Automatically adjusting the number of retrieved chunks (k) based on a rolling 5-query sentiment average.

Why vs Alternative: A static k=5 either overwhelms a user who understands the topic, or starves a confused user of necessary context. The adaptive approach solves this.

Before → After: Fixed context → Dynamic context (confused: base_k + 3, neutral: base_k, clear: max(2, base_k - 2)).

Why it mattered: This is the adaptive core thesis of the project. It transforms the pipeline from a static search engine into an empathetic tutor that provides detailed, step-by-step context when needed, and concise answers when the student is tracking.

5. Corrective RAG (CRAG) Guardrails

What: A three-stage agentic pipeline (Rewrite → Grade → Generate/Verify) via raw Ollama API calls.

Why vs Alternative: Standard RAG blindly feeds retrieved chunks to the LLM. CRAG grades chunk relevance and verifies groundedness. Used raw API calls instead of LangChain to keep the dependency footprint light.

Before → After: Direct generation → Guarded pipeline with explicit toggles (RAG_CRAG_ENABLED).

Why it mattered: It never silently presents unsupported content as fact. If retrieved chunks are irrelevant, the system catches it and adjusts or adds a disclaimer, significantly reducing hallucinations.

6. Sync CLI to Async FastAPI & Startup-Once Loading

What: Re-architecting into an async REST API and loading heavy resources (models, indices) once in app/core/lifespan.py.

Why vs Alternative: Re-loading models per request adds 3-4s of overhead. A sync server blocks the event loop for 2-10s per LLM call, preventing concurrent users.

Before → After: 3-4s startup per query → ~0ms resource load overhead per query. Blocking CLI → Concurrent async HTTP server.

Why it mattered: Essential for scaling from a prototype to a production API that can serve multiple simultaneous users and health-check monitors without freezing.

7. Structured Logging + Request Tracing

What: Python logging with JSON format and a UUID request_id threaded via contextvars.

Why vs Alternative: Unstructured print statements are useless in a concurrent environment where logs interleave.

Before → After: print() statements → Machine-readable JSON logs with timestamp, level, request_id, and stream tags.

Why it mattered: Allows reconstructing the exact lifecycle of a single request in production, making debugging and operational monitoring possible.

8. Embedding Cache

What: An LRU cache (256 entries) keyed by the SHA-256 hash of the query text.

Why vs Alternative: Re-embedding the exact same query (common during CRAG re-retrieval or repeated student questions) is wasted compute.

Before → After: Embedding every time → Serving from cache. Saves ~{{ "%.3f" | format(data.benchmarks.embedding_latency_mean_ms) if data.benchmarks else "21.59" }}ms per cache hit.

Why it mattered: Lowers CPU/GPU load and decreases total latency for repeated operations without adding external dependencies like Redis.

9. Containerization

What: A multi-stage Dockerfile and docker-compose.yml for isolated deployment.

Why vs Alternative: Relying on the host OS leads to "works on my machine" bugs, especially with C++ dependencies like FAISS and libmagic.

Before → After: Manual pip install → Single docker-compose up with isolated environments and volume mounts for persistence.

Why it mattered: Ensures reproducible, standardized deployments across local testing, staging, and production environments.
